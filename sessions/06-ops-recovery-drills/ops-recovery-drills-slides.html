R5bsnapshot rollback확장 · rebuild vs rollback
R5의 두 번째 복구. 원천 재생성 대신 Iceberg snapshot 포인터를 직전 정상으로 즉시 되돌린다.

복구 B — rollback (즉시)

# find-recovery-point가 준 snapshot_id 사용
./scripts/ops-r5b-rollback-iceberg-mart.sh <snapshot_id>
./scripts/query-bi-metrics.sh

0 → 759 즉시 (Spark·StarRocks)

포인트 — 두 복구의 트레이드오프

  • rebuild (R5 · Airflow): 원천(Paimon)에서 재derive. 권위 있는 정답, 느림, 원천 정상 필요.
  • rollback (R5b): metadata 포인터만 과거로. 즉시, 단 그 snapshot 이후 commit을 전부 폐기 → "그래도 되는 상황"에서만.
  • find-recovery-point = "어느 snapshot까지 정상인가" 확정 → rollback = 그 다음 결정·실행.
R6StarRocks metadata refresh조회 계층 검증
원천은 정상인데 조회 계층이 최신을 못 보는 경우. 독립 파괴가 아니라 serving 계층 검증 라운드.

관찰 / 복구

./scripts/query-bi-metrics.sh           # 전
./scripts/ops-r6-refresh-starrocks-iceberg.sh
./scripts/query-bi-metrics.sh           # 후

포인트

이 스택은 조회 시 catalog를 재생성해 거의 항상 fresh.

refresh 전후 숫자가 같으면 정상("이미 fresh"). 실패가 아니라, "조회 계층은 원천과 별도로 확인한다"는 교훈은 동일. native·StarRocks·BI 3자 비교.
