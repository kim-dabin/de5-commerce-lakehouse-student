Part A · 데이터 품질 (Data Quality)

데이터 품질을 표준 차원으로 나누어, 규칙 목록 자체가 문서이자 게이트가 되도록 Great Expectations로 선언합니다. 5차시 DAG의 validate_bi_metrics()는 같은 검증을 파이썬 assert로 코드에 묻어 두었지만, 여기서는 같은 의도를 선언형 스위트로 승격합니다.

A-1품질 5차원 + 전체 검증 실행검증됨

개념 · 5가지 품질 차원

차원질문도구
completeness빠진 게 없는가 (row count, not-null)GE
uniqueness키가 중복되지 않는가GE
validity값이 허용 범위/집합 안인가 (rating 1~5, event_type 7종)GE
consistency컬럼 간 관계가 맞는가 (퍼널 단조성, 집계 합)Spark
freshness최신 데이터인가Spark

completeness/uniqueness/validity는 GE로 선언하고, GE 기본 expectation에 없는 consistency/freshness는 Spark로 직접 계산합니다.

A-2의도적 실패 데모 — 검증이 없으면 조용히 통과한다검증됨

의도 · 실무 사례 N (무음 드롭)

"잡이 실패하지 않음"이 곧 "데이터가 옳다"는 뜻은 아닙니다. 검증이 없으면 오염은 BI까지 조용히 흘러갑니다. 정상 데이터를 메모리상 복사한 뒤 한 행만 오염시켜, 같은 스위트가 잡아내는지 봅니다. 실제 Iceberg 테이블은 건드리지 않습니다.

오염 주입 (노트북 §3)

# 정상 category_daily(759행) 복사 → 한 행만 오염
bad_row = good.limit(1) \
  .withColumn("category_code", F.lit(None))   # not-null 위반
  .withColumn("event_count", F.lit(0))        # between(>=1) 위반
bad = good.unionByName(bad_row)               # 760행 → row_count도 위반

검출 — DQ가 오염을 잡아냄

overall success: False   # False가 정상 (DQ가 잡았다는 뜻)
expect_table_row_count_to_equal       FAIL  observed=760
expect_column_values_to_not_be_null   FAIL  column=category_code  unexpected=1
expect_column_values_to_be_between    FAIL  column=event_count    unexpected=1

포인트

이 오염이 BI에 들어가면 카테고리 없는 행/0 매출이 대시보드를 흔듭니다. 품질 게이트가 있으면 serving 전에 막힙니다. 검증의 가치는 "통과할 때"가 아니라 "잡아낼 때" 드러납니다.

Part B · StarRocks Serving / Realtime OLAP / BI

StarRocks는 Paimon·Iceberg 데이터를 복사하지 않고 external catalog로 직접 조회하는 serving/query 계층입니다. 같은 Olist 데이터를 두 질문으로 나눕니다.

Paimon ux/review/order (current-state)  ─StarRocks Paimon catalog→  Realtime OLAP  "지금 무슨 일이?"
Iceberg analytics mart (L1/L2)          ─StarRocks Iceberg catalog→ Daily Business BI "믿을 수 있는 기준은?"
B-1external catalog로 직접 조회serving

실행

./scripts/query-realtime-olap.sh             # Paimon external catalog 대표 쿼리

포인트 — external catalog vs internal table

StarRocks는 Paimon/Iceberg를 internal table로 복사할 수도 있고, external catalog로 원본을 직접 조회할 수도 있습니다. 이 스택은 external catalog로 원본을 그대로 조회합니다 — 복사 지연/이중관리가 없는 대신, 원본 포맷·메타데이터에 조회 성능이 묶입니다. 언제 복사(internal)하고 언제 직접 조회(external)할지가 serving 설계 판단입니다.
B-2realtime OLAP vs batch BIserving

의도 · 같은 데이터, 두 질문

Realtime OLAP은 Paimon current-state로 "지금 무슨 일이 일어나는가"를, Batch BI는 Iceberg mart로 "믿을 수 있는 기준 결과"를 답합니다.

실행

./scripts/query-realtime-olap-metrics.sh     # Paimon current-state → realtime
./scripts/query-bi-metrics.sh                # Iceberg mart → batch 기준

관찰 결과 — realtime 대표 지표:

total_events 16,693 · users 2,875 · sessions 2,875 · products 1,470
revenue 265,036 · event_types 7   # purchase 2,220건이 매출 전부를 만든다

포인트

같은 데이터라도 Paimon current-state(realtime)와 Iceberg mart(batch)는 grain/시점이 다릅니다. 숫자가 다르면 "어느 테이블·어느 grain에서 계산됐는가"를 먼저 확인합니다. UXLog의 purchase는 행동 이벤트입니다 — 공식 매출 지표로 쓰려면 주문/결제/환불 데이터가 더 필요합니다.

포인트 — 사례 K (뷰 의존성)

데이터 파일과 catalog가 정상이어도 조회 계층 metadata가 stale할 수 있습니다. 대시보드/알림도 upstream table·view 의존성을 갖는 파이프라인입니다. 이 스택은 조회 시 catalog를 재생성해 거의 항상 fresh하므로, refresh 전후 숫자가 같으면 "이미 fresh"이며 실패가 아닙니다. 가능하면 native count · StarRocks count · BI metric을 3자 비교합니다.
