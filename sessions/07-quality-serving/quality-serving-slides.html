포인트 — external vs internal

  • external: 원본 직접 조회. 복사 지연·이중관리 없음. 성능은 원본 포맷에 묶임.
  • internal: StarRocks로 복사. 빠른 serving, 대신 적재/동기화 필요.
  • 언제 복사하고 언제 직접 조회할지가 serving 설계 판단.
B-2realtime OLAP vs batch BI같은 데이터, 두 질문
Paimon current-state는 "지금 무슨 일이?", Iceberg mart는 "믿을 수 있는 기준은?"에 답한다.

포인트

데이터/카탈로그가 정상이어도 조회 계층 metadata가 stale할 수 있다. native · StarRocks · BI metric을 3자 비교한다.

왜 이 순서인가
품질 게이트 → serving

한 줄 요약

  • 품질이 통과해야 serving 숫자를 신뢰할 수 있다.
  • 품질은 규칙으로 선언(GE) + 관계/시간은 직접 계산(Spark).
  • serving은 같은 데이터를 realtime/batch 두 질문으로 나눈다.
  • 조회 계층도 별도로 검증한다(뷰 의존성).

실무 사례 연결

  • N · 무음 드롭: 검증 없으면 오염이 조용히 통과 → A-3 실패 데모.
  • K · 뷰 의존성: 대시보드도 upstream 의존 → B-3 refresh.

잡이 실패하지 않음이 곧 데이터가 옳다는 뜻은 아니다.

증거로 남기기 & 과제

수업에서 남길 증거

  • 품질: 어떤 차원의 어떤 규칙 (PASS/FAIL)
  • 실패 데모: 어떤 오염 → 어떤 expectation이 (unexpected 몇으로) 잡음
  • serving: realtime 1개 + batch 1개를 어느 grain에서 읽었나

과제 (발표 자료로 축적)

  • A. 품질 규칙 1개 추가 → PASS와 FAIL 둘 다 캡처
  • B. realtime/batch 지표 1개씩 grain까지 설명
  • C. 리스크·개선 1개 + 발표용 한 문장
