DE5 Session 05 · Spark to Iceberg Batch BI

마트 테이블 설계 가이드

이 문서는 오늘 만든 Iceberg Analytics Mart가 왜 7개 테이블로 나뉘었는지, 각 테이블이 어떤 질문에 답하는지, 그리고 BI가 이 테이블을 어떻게 사용하는지 설명합니다.

핵심 문장

마트 설계는 테이블을 많이 만드는 일이 아니라, 어떤 질문에 빠르게 답할 수 있도록 기준 데이터를 어떤 grain으로 남길지 정하는 일입니다.

UX 행동 로그는 event-level fact로 남깁니다.
리뷰와 주문은 current-state entity로 정리합니다.
일별/카테고리별/퍼널별로 자주 볼 질문은 aggregate mart로 미리 만듭니다.

오늘의 설계 한눈에 보기

Paimon은 수집 결과와 현재 상태를 저장하고, Spark는 이를 읽어 Iceberg에 분석용 mart를 만듭니다. StarRocks는 Iceberg external catalog로 이 mart를 조회하고, Streamlit은 그 결과를 화면으로 보여줍니다. 

Paimon Bronze/current
  -> Spark batch transform
  -> Iceberg analytics mart
  -> StarRocks external catalog
  -> Streamlit Daily Business BI

2. 입력 데이터: Paimon 3개 테이블

Spark가 읽는 입력은 Paimon에 저장된 3개 테이블입니다. 여기서 이미 append와 upsert 모델이 나뉩니다.

Paimon table 성격 의미 왜 이렇게 저장하나
ux_events_bronze append fact 사용자가 한 행동 로그 같은 사용자가 같은 상품을 두 번 보면 두 번의 행동이므로 두 row가 맞습니다.
review_current upsert current-state 리뷰의 최신 상태 같은 review_id에 대해 매칭/감성/사용 여부가 나중에 바뀔 수 있어 최신 상태가 중요합니다.
order_current upsert current-state 주문의 최신 상태 주문은 created, approved, delivered 같은 상태 변경을 거치므로 order_id 기준 최신 상태가 필요합니다.

append fact: 발생 사실을 계속 쌓는다

  • 질문: 사용자가 무엇을 했는가?
  • 예: 상품 조회, 리뷰 노출, 장바구니, 구매 행동
  • 같은 행동이 반복되면 row도 반복됩니다.

current-state entity: 최신 상태를 유지한다

  • 질문: 지금 이 리뷰/주문은 어떤 상태인가?
  • 예: 리뷰 감성, 리뷰 사용 여부, 주문 상태
  • 같은 key의 변경 이벤트는 한 row로 접힙니다.

2-1. L0/L1/L2 계층 점검

이번 프로젝트는 "raw를 그대로 쌓고 끝"이 아니라, 같은 데이터를 목적별 계층으로 나눠 읽습니다. 수업에서는 아래처럼 L0 수집 계층, L1 기준 계층, L2 집계/서빙 계층으로 설명하면 됩니다.

계층 테이블 grain 저장 모델 검증 포인트
L0
Paimon Bronze/current		 ux_events_bronze event_id append fact, raw_json 보관, bucket=3 Kafka ux-events 16,693건과 row count가 맞는가?
review_current review_id primary key upsert, 무파티션, bucket=3 review-events 5,943건이 review_id 기준 1,971건으로 접히는가?
order_current order_id primary key upsert, 무파티션, bucket=3 order-status-events 7,886건이 order_id 기준 2,000건으로 접히는가?
L1
Iceberg base mart		 olist_ux_events_clean event_id 정제 fact, partition=event_date timestamp/date/category/brand/price 보정 후에도 16,693건인가?
olist_review_current review_id current-state 기준 테이블, 무파티션 category_code는 필터 속성일 뿐 파티션 기준으로 쓰지 않는다.
olist_order_current order_id current-state 기준 테이블, 무파티션 주문 엔티티 최신 상태 2,000건이 유지되는가?
L2
Iceberg aggregate/query mart		 olist_event_type_daily event_date + event_type 일별 이벤트 타입 집계, partition=event_date 날짜별 유입/행동량을 빠르게 볼 수 있는가?
olist_funnel_daily event_date 일별 funnel 집계, partition=event_date 상품 조회, 리뷰 노출, 장바구니, 구매 흐름을 설명할 수 있는가?
olist_category_daily event_date + category_code 카테고리별 행동/매출 집계, partition=event_date 자주 보는 category 필터는 current 테이블이 아니라 집계 mart에서 처리한다.
olist_review_sentiment_by_category category_code + sentiment 카테고리별 리뷰 감성 집계 부정 리뷰가 몰린 카테고리를 빠르게 찾을 수 있는가?

실무적으로는 review/order의 변경 이력도 review_events_bronze, order_status_events_bronze 같은 append L0 테이블로 보존하는 것이 더 안전합니다. 현재 MVP는 streaming upsert와 current-state 해석을 먼저 보여주기 위해 current 테이블부터 만들었습니다.

3. 출력 데이터: Iceberg Analytics Mart 7개

Spark는 Paimon 3개 테이블을 읽고 Iceberg analytics namespace에 7개 mart를 만듭니다. 설계는 크게 기준 테이블 3개와 집계 테이블 4개로 나뉩니다.

base mart

기준 테이블

BI 지표가 이상할 때 다시 돌아가서 검증할 수 있는 정제 기준 데이터입니다.

aggregate mart

집계 테이블

일별, 카테고리별, 퍼널별로 자주 보는 질문에 빠르게 답하기 위한 요약 결과입니다.

query layer

BI 계산

리뷰 영향처럼 조합이 필요한 지표는 StarRocks/Spark 쿼리 계층에서 추가 계산합니다.

Iceberg mart grain 목적 대표 질문
olist_ux_events_clean 이벤트 1건 = 1 row UXLog 정제 fact table 전체 이벤트 수, 사용자 수, 세션 수, 구매 행동 금액은?
olist_review_current 리뷰 1개 = 1 row 리뷰 최신 상태 기준 테이블 현재 리뷰 수, 평균 평점, 부정 리뷰 수는?
olist_order_current 주문 1개 = 1 row 주문 최신 상태 기준 테이블 현재 주문 엔티티 수와 주문 상태 분포는?
olist_event_type_daily 날짜 + 이벤트 타입 일별 이벤트 타입 집계 날짜별로 어떤 이벤트가 얼마나 발생했나?
olist_funnel_daily 날짜 일별 퍼널 집계 상품 조회, 리뷰 노출, 장바구니, 구매 세션이 어떻게 이어졌나?
olist_category_daily 날짜 + 카테고리 카테고리별 행동/매출 집계 어떤 카테고리에서 조회, 구매, 매출이 높았나?
olist_review_sentiment_by_category 카테고리 + 감성 카테고리별 리뷰 감성 집계 어떤 카테고리에 부정 리뷰가 많이 몰렸나?

4. 각 테이블을 왜 이렇게 만들었나

4-1. olist_ux_events_clean: 행동 로그의 기준 fact

Paimon의 ux_events_bronze를 분석 가능한 형태로 정리한 테이블입니다. timestamp와 date를 만들고, category/brand/price의 비어 있는 값을 보정합니다. 

grain: event_id
partition: event_date
primary question: 사용자가 언제, 어떤 상품에, 어떤 행동을 했는가?

4-2. olist_review_current: 리뷰 최신 상태 기준

Paimon의 review_current를 Iceberg 기준 테이블로 옮깁니다. 리뷰 감성, 평점, 사용 여부, 매칭 상품 정보를 함께 둡니다. 

grain: review_id
partition: 없음
primary question: 지금 기준으로 상품/카테고리에 어떤 리뷰 상태가 존재하는가?

category_code는 리뷰를 해석하고 필터링할 때 중요한 속성이지만, 리뷰 엔티티의 최신 상태 테이블을 물리적으로 나누는 기준으로는 쓰지 않습니다. 카테고리 기준으로 자주 보는 질문은 olist_category_daily, olist_review_sentiment_by_category, 또는 StarRocks/Spark query layer에서 처리합니다.

4-3. olist_order_current: 주문 최신 상태 기준

주문 상태 이벤트는 여러 번 발생할 수 있지만, 분석 기준으로는 주문의 최신 상태가 필요합니다. 그래서 order_id 기준 현재 상태를 mart로 남깁니다. 

grain: order_id
primary question: 지금 기준으로 주문 엔티티는 어떤 상태인가?

4-4. 집계 mart 4개: BI에서 자주 보는 질문을 미리 계산

매번 event-level fact를 처음부터 group by 해도 되지만, 수업에서는 BI 해석을 명확히 보여주기 위해 일별/카테고리별/퍼널별 대표 집계를 Iceberg mart로 남겼습니다.

6. 숫자가 다르면 바로 유실인가?

아닙니다. 먼저 테이블의 grain과 저장 모델을 확인해야 합니다.

예를 들어 review-events는 Kafka에 5,943건이 들어가지만, review_current는 1,971건입니다. 이것은 유실이 아니라, 같은 review_id의 상태 변경 이벤트가 current-state 테이블에서 최신 상태 한 줄로 접힌 결과입니다. 

Kafka review-events        5,943 events
Paimon review_current      1,971 rows
Iceberg olist_review_current 1,971 rows

의미: 이벤트 수와 엔티티 수는 다를 수 있다.

또 하나의 예시는 orders와 order_current_rows입니다.

지표 값 의미
BI orders 1,968 UX 이벤트에서 purchase/order-linked 기준으로 집계한 주문 수입니다.
order_current_rows 2,000 주문 엔티티 current-state 테이블의 row count입니다.

두 값이 다른 이유는 파이프라인 실패가 아니라 지표 정의가 다르기 때문입니다.

7. BI는 이 mart를 어떻게 쓰나

Streamlit BI는 직접 파일을 읽지 않습니다. StarRocks가 Iceberg external catalog로 mart를 조회하고, Streamlit은 그 결과를 받아 차트로 보여줍니다. 

Streamlit Daily Business BI
  -> query-bi-metrics.sh
  -> StarRocks Iceberg external catalog
  -> Iceberg analytics mart
  -> BI_METRICS_JSON
1
상단 KPI

전체 이벤트, 사용자, 세션, purchase 행동 금액을 봅니다.

2
Daily Revenue and Conversion Trend

일자별 매출과 전환 흐름을 봅니다. Iceberg mart는 회고와 재처리에 유리합니다.

3
Batch Funnel Sessions

상품 조회, 리뷰 노출, 리뷰 확장, 장바구니, 구매 세션을 funnel로 봅니다.

4
Trusted Review Impact

리뷰를 본 사용자가 장바구니/구매로 이어졌는지 batch 기준으로 다시 계산합니다.

5
Category Business Health

매출, 리뷰 커버리지, 부정 리뷰 비율, 이탈률을 함께 보며 점검 후보를 찾습니다.

8. 수업에서 가져갈 설계 원칙

grain

먼저 row의 의미를 정한다

이 row가 이벤트 1건인지, 리뷰 1개인지, 날짜+카테고리 집계인지 먼저 정해야 합니다.

model

append와 upsert를 구분한다

발생 사실은 append, 최신 상태가 중요한 엔티티는 upsert/current-state로 봅니다.

proof

숫자는 정의와 함께 본다

count가 다르면 유실이라고 단정하지 말고, 입력 이벤트 수와 출력 grain을 비교합니다.

발표에서 쓰기 좋은 문장

이 프로젝트는 UXLog를 event-level fact로 정리하고,
리뷰/주문은 current-state entity로 유지한 뒤,
Spark batch로 Iceberg Analytics Mart를 만들어
리뷰 노출 이후의 전환과 부정 리뷰 기반 이탈 리스크를 분석합니다.

9. 실무 보강 로드맵

오늘의 목표는 MVP를 end-to-end로 완성하는 것입니다. 다만 실무형 포트폴리오라면 "현재 구현의 한계와 다음 개선 순서"까지 설명할 수 있어야 합니다. 아래 항목은 오늘 반드시 구현해야 하는 내용이 아니라, 6차시 운영 검증과 이후 고도화에서 다룰 로드맵입니다.

좋은 설계 문서는 완벽한 척하지 않습니다. 현재 상태, 알려진 갭, 실무 패턴, 적용 우선순위를 함께 남깁니다.

보강 지점 현재 구현 실무 패턴 적용 시점
검증 전 퍼블리시 DROP TABLE -> CREATE TABLE AS SELECT -> validate 순서라, 잘못된 결과가 먼저 노출될 수 있습니다. Write-Audit-Publish. staging/branch에 먼저 쓰고 검증 통과 후 publish합니다. 차선책은 INSERT OVERWRITE로 빈 테이블 구간을 줄입니다. 6차시 전후 운영 검증 실습
review/order 이벤트 이력 UX는 append bronze가 있지만 review/order는 바로 current-state로 접힙니다. review_events_bronze, order_status_events_bronze를 append로 저장하고, current table은 그 이력에서 파생합니다. 기수 이후 고도화 또는 심화 과제
current table 파티션 olist_review_current는 무파티션 current-state 기준 테이블입니다. 카테고리 조회는 current table 파티션이 아니라 집계 mart나 query layer에서 처리합니다. 데이터가 커지면 id 기반 bucketing을 검토합니다. 반영 완료
상수 count 검증 샘플 기준 16,693, 1,971, 2,000 같은 고정 기대값을 검증합니다. 관계 불변식을 함께 검증합니다. 예: rows(ux_clean) = rows(ux_bronze), distinct review_id = rows(review_current). 6차시 운영 검증에서 추가하기 좋음
증분/백필 매 run 전체 mart를 재생성합니다. 날짜 파라미터와 rolling window overwrite로 최근 N일을 재계산하고, late event를 흡수합니다. 프로덕션 고도화 로드맵
리뷰 감성 시간 축 review_sentiment_by_category는 전 기간 current-state 기준 누적 집계입니다. as_of_date 또는 snapshot 기준을 추가해 시간에 따른 변화와 소급 변경을 추적합니다. BI 고도화
시간대와 funnel 귀속 규칙 event_date는 Spark session timezone인 UTC 기준입니다. funnel 단계 귀속 규칙은 코드에만 존재합니다. 비즈니스 시간대와 funnel 단계 인정 조건을 문서화합니다. 예: America/Sao_Paulo 기준 일자, 이벤트 순서/단계 건너뜀 처리. 문서에는 즉시 명시, 지표 정의서로 확장
StarRocks metadata refresh Iceberg table을 drop/recreate하면 external catalog 메타데이터 캐시가 stale해질 수 있습니다. build 후 REFRESH EXTERNAL TABLE 또는 catalog refresh task를 둡니다. WAP/overwrite 구조로 바꾸면 리스크가 줄어듭니다. 6차시 장애/운영 검증 소재
