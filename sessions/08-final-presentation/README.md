# 8차시 - 최종 발표 (포트폴리오)

마지막 차시는 새 코드를 만드는 시간이 아니라, 1~7차시에 함께 만든 공통 프로젝트를 **각자의 관점으로 해석·검증해 포트폴리오 발표로 완성**하는 시간입니다.

> 핵심 문장: "저는 이 파이프라인을 **실행**했습니다"가 아니라,
> "이 파이프라인이 **어떤 질문에 답하고, 어디까지 정상인지 어떤 증거로 확인했으며, 남은 갭을 어떻게 개선할지** 설명할 수 있습니다."

## 이 폴더 사용법

| 파일 | 용도 | 언제 |
|---|---|---|
| `final-presentation-guide.html` | 발표 방향·구성·검증 체크리스트·보강 로드맵 (전체 안내) | 가장 먼저 정독 |
| `final-presentation-slides.html` | 위 안내를 한 화면씩 보는 개요 슬라이드 (←/→ 이동) | 빠르게 훑을 때 |
| `EVIDENCE_CARD.md` | 발표 증거를 스택에서 직접 수집하는 명령 모음 | 발표 자료 만들기 전 |
| `PRESENTATION_TEMPLATE.md` | 7장 발표 슬라이드 빈칸 템플릿 (복사해서 채움) | 슬라이드 작성 시 |
| `ASSIGNMENT.md` | 최종 제출물 요건과 평가 기준(루브릭) | 제출 전 점검 |

## 발표가 모두 같으면 안 됩니다

공통 기반(Olist + UX Events, Kafka → Flink → Paimon → Spark/Iceberg → StarRocks/BI → Airflow)은 같지만, **질문 · 지표 해석 · 검증 증거 · 운영 리스크 · 개선 로드맵**은 각자 달라야 포트폴리오로 의미가 있습니다.

## 준비 흐름

```text
1. final-presentation-guide.html 정독 — "질문 → 구조 → 증거 → 해석 → 개선" 흐름 이해
2. 내 질문 1개 선택 (예: 리뷰를 본 사용자의 구매 전환 / 부정 리뷰와 이탈)
3. EVIDENCE_CARD.md로 내 질문에 필요한 계층별 증거 수집·캡처
4. PRESENTATION_TEMPLATE.md를 복사해 7장 슬라이드 채우기
5. ASSIGNMENT.md 루브릭으로 자가 점검 후 제출
```

증거를 수집하려면 5차시 스택이 떠 있어야 합니다.

```bash
cd sessions/05-project-mvp
docker compose -f docker-compose.lite.yml up -d
```

## 발표일 진행 (참고)

```text
1인 발표 + 짧은 Q&A.
순서: 질문 → 아키텍처 → 데이터/지표 → 계층별 증거 → 리스크/개선 → 한 문장 마무리.
"돌아갔다"가 아니라 "어떤 증거로 정상이라고 판단했는가"를 중심에 둡니다.
```

## 그동안 쌓은 과제가 곧 재료입니다

6차시(장애 대응 인시던트 노트)와 7차시(품질 규칙 + serving 해석) 과제가 그대로 발표의 "검증 증거"와 "리스크/개선" 슬라이드 재료가 됩니다. 새로 만들기보다 **쌓아온 것을 묶으세요.**
