Spark UI · 실행 플랜 읽기

UI 화면 소개가 아니라 현업에서 실제로 보는 법입니다 — 잡이 느릴 때 어디를 보는지, stage·task에서 어떤 이상 신호를 잡는지, 실패하거나 리소스를 많이 먹을 때 원인을 어떻게 좁히는지. 모든 명령과 플랜은 실제 스택에서 검증한 값입니다.

"느리거나 실패하면 — 메모리부터 늘리지 말고, 플랜의 셔플 수 · stage의 task 편향 · spill부터 본다."

0. Spark UI 여는 법

spark-client에서 Spark job이 돌면 Spark UI가 4040에 뜹니다. UI는 driver가 살아있는 동안만 보입니다 — 한 번에 안 뜨면 job이 이미 끝난 것입니다.

# 로컬이면 브라우저에서
http://localhost:4040
# 원격(headless) 호스트면 SSH 포워딩 후 접속
ssh -L 4040:localhost:4040 <host>
오래 들여다보려면 대화형 세션(아래 1번)을 띄워 두세요. 끝난 job을 나중에 보려면 History Server(localhost:18080, 아래 카드 2.5) — 이 스택에 추가했습니다.
1실행 플랜 읽기 — EXPLAIN검증됨

실행

cd sessions/05-project-mvp
./scripts/run-spark-iceberg-sql.sh labs/10-spark-ui/explain_category_daily.sql

실제 출력 — category_daily 집계의 물리 플랜

== Physical Plan ==
Sort (6)
+- Exchange (5)        rangepartitioning(...)      ← ORDER BY 용 셔플
   +- HashAggregate (4)   final                     ← 최종 집계
      +- Exchange (3)   hashpartitioning(..., 200)  ← GROUP BY 용 셔플 (파티션 200)
         +- HashAggregate (2)  partial             ← map-side 부분 집계
            +- BatchScan olist_ux_events_clean (1)     ← Iceberg 스캔
+ AdaptiveSparkPlan (AQE)

읽는 법

  • 아래에서 위로 읽는다: 스캔 → 부분집계 → 셔플 → 최종집계 → (정렬 셔플) → Sort.
  • Exchange = 셔플 = 비싼 구간. 여기선 2번(GROUP BY + ORDER BY). ORDER BY를 빼면 셔플 1번을 아낀다.
  • hashpartitioning(…, 200) = shuffle partition 기본값 200. 작은 데이터엔 빈 task 200개 = 과분할 신호 → spark.sql.shuffle.partitions 조정 대상.
  • AdaptiveSparkPlan(AQE) = 런타임에 파티션 합치기/조인 전략 변경. isFinalPlan=false는 실행 전 플랜이라는 뜻.
  • BatchScan에 PushedFilters가 보이면 = predicate pushdown(스캔에서 미리 거름)이 먹은 것.

탭별로 무엇을 보나

탭볼 것 / 이상 신호
SQL / DataFrame쿼리별 플랜 + 노드별 소요시간·행수. 어느 노드가 오래 걸리나.
Stages느린 stage 클릭 → Task 분포. duration max ≫ median = skew(편향). Shuffle Read/Write 큰 stage = 셔플 병목. Spill(Memory/Disk) = 메모리 부족으로 디스크에 흘림 = 느림.
Jobs어느 job/stage에 시간이 쌓이나(전체 그림).
2.5History Server — 끝난 잡 다시 보기localhost:18080

왜 필요한가

4040은 driver가 사는 동안만 뜨고, 이 스택은 데이터가 작아 잡이 수 초면 끝나 창을 열 새가 없습니다. History Server는 spark.eventLog로 남긴 기록을 읽어 끝난 잡의 UI(SQL plan·Stages·Task·skew)를 나중에 차분히 보여줍니다 — 데모·디버깅엔 이게 4040보다 실용적입니다.

보기

# §2 명령처럼 eventLog를 켜고 잡을 돌린 뒤
http://localhost:18080        # 완료된 application 목록 → 클릭하면 4040과 동일한 UI

포인트

잡에 spark.eventLog.enabled=true + spark.eventLog.dir=file:///workspace/data/spark-events가 있으면 자동 기록됩니다. 잡이 너무 빨라 4040을 놓쳤어도 18080에서 같은 plan·stages·task를 봅니다.
3실패하거나 리소스를 많이 먹을 때R8 데모

데모 (멘토) — collect 직전 일부러 멈춰 UI 볼 시간을 준다

OOM_UI_PAUSE_SECONDS=40 ./scripts/ops-r8-spark-driver-oom-demo.sh
# pause 동안 Executors 탭: driver/executor 메모리·GC time·active task
# 이후 collect()로 큰 결과를 driver로 모으면 →
java.lang.OutOfMemoryError: Java heap space

원인 좁히기

  • 증상 신호: OOM, GC time 과다(연산보다 GC가 김), executor lost.
  • 복구의 시작은 "메모리 늘리자"가 아니라 "왜 그렇게 많이 필요했나" → collect() 회피 / limit·집계로 줄이기 / skew 점검 / shuffle partition 조정.
  • OOM은 이 spark-submit JVM 안에 갇혀 다른 컨테이너에는 영향이 없다(자원 격리).

한 장 체크리스트

상황어디부터 보나
느릴 때SQL탭 느린 노드 → Stages 탭 task skew(max vs median) → Shuffle Read/Write · Spill
실패할 때에러 로그(OOM / GC / executor lost) → Executors 탭 → driver냐 executor냐
리소스 과다collect() / 큰 셔플 / skew / 과분할(shuffle partition 200) 의심
플랜 볼 때아래→위로 읽기 · Exchange(셔플) 수 세기 · PushedFilters 확인 · AQE 여부
