최종 발표 안내 — 공통 프로젝트를 개인 포트폴리오로 정리하기

이번 최종 발표는 각자 완전히 다른 데이터를 새로 찾는 방식이 아니라, 우리가 함께 만든 Olist 기반 실무형 데이터 파이프라인을 각자의 관점으로 해석하고 검증해 포트폴리오로 완성하는 방식입니다.

공통 주제 / 공통 데이터 / 공통 파이프라인을 기반으로 하되, 최종 발표는 각자의 비즈니스 질문, 지표 해석, 운영 검증, 개선 로드맵으로 차별화합니다.

공통 기반: Olist 커머스 데이터 + 사용자 행동 이벤트
공통 파이프라인: Kafka → Flink → Paimon → Spark/Iceberg → StarRocks/BI → Airflow
개인화 지점: 질문, 지표, 증거, 리스크, 개선 방향

최종 발표의 핵심 문장

“저는 이 파이프라인을 실행했습니다”가 아니라, “이 파이프라인이 어떤 질문에 답하고, 어디까지 정상인지 어떤 증거로 확인했으며, 남은 운영 갭을 어떻게 개선할지 설명할 수 있습니다”가 목표입니다.

완벽한 시스템보다, 완성한 것과 남은 갭을 구분하고 우선순위를 말할 수 있는 발표가 더 실무형입니다.

1. 우리가 공통으로 완성한 것

아래 흐름은 모든 수강생이 공유하는 기본 골격입니다. 발표에서는 이 전체를 다 깊게 설명하기보다, 본인이 선택한 질문과 지표가 이 흐름의 어느 지점에서 만들어졌는지 연결해서 설명하면 됩니다.

Olist + UX Events주문 · 리뷰 · 행동 이벤트
Kafkatopic 3개 · replay
Flinkstreaming jobs 3개
PaimonL0 · append/upsert

StarRocks/BIOLAP 조회 · 대시보드
Data

데이터

Olist 주문/리뷰 데이터와 사용자 행동 이벤트를 결합해 “리뷰와 구매 행동의 관계”를 분석할 수 있는 시나리오를 만듭니다.

Pipeline

파이프라인

Kafka 수집, Flink streaming 적재, Paimon L0 저장, Spark/Iceberg mart 생성, StarRocks/BI 조회까지 연결합니다.

Operation

운영 검증

Airflow DAG와 task log를 통해 “어디까지 정상인지”를 계층별 증거로 확인합니다.

2. 모두 같으면 안 되는 것

기본 프로젝트는 공통이지만, 최종 포트폴리오의 메시지는 달라야 합니다. 같은 대시보드를 보더라도 어떤 질문을 잡고, 어떤 지표를 해석하고, 어떤 운영 리스크를 발견했는지에 따라 발표는 충분히 달라질 수 있습니다.

Question

비즈니스 질문

리뷰를 본 사용자가 구매로 이어졌는가? 부정 리뷰가 많은 카테고리는 이탈이 높은가? 본인이 선택한 질문을 명확히 둡니다.

Metrics

지표 해석

BI 숫자를 그대로 읽지 말고, 어떤 테이블과 grain에서 계산된 지표인지 설명합니다.

Evidence

검증 증거

Kafka, Flink, Paimon, Iceberg, StarRocks, Airflow 중 어디에서 어떤 증거를 봤는지 남깁니다.

Risk

운영 리스크

count 불일치, append/upsert 차이, cache stale, reset 후 재빌드, 이벤트 이력 소실 같은 리스크를 본인 언어로 정리합니다.

Roadmap

개선 방향

검증 게이트, 이벤트 bronze 추가, 관계 기반 검증, StarRocks refresh task, WAP 같은 다음 단계를 제안합니다.

Story

나의 관점

“이 프로젝트에서 내가 가장 중요하게 본 문제는 무엇인가”를 하나의 메시지로 정리합니다.

3. 발표 추천 구성

아래 구성은 권장안입니다. 발표 시간이 정해지면 장수를 줄이되, 최소한 “질문 → 구조 → 증거 → 해석 → 개선” 흐름은 유지하는 것이 좋습니다.

순서 슬라이드 넣어야 할 내용 핵심 메시지
1 프로젝트 한 줄 소개 Olist 기반 커머스 행동/리뷰/주문 데이터를 이용해 lakehouse + OLAP 파이프라인을 구축했다고 소개합니다. 무엇을 만든 프로젝트인가
2 내가 선택한 질문 “리뷰를 본 사용자의 구매 전환”, “부정 리뷰와 이탈”, “append/upsert 설계” 등 본인이 집중할 질문을 선언합니다. 왜 이 프로젝트가 필요한가
3 전체 아키텍처 Kafka → Flink → Paimon → Spark/Iceberg → StarRocks/BI → Airflow 흐름을 그림으로 보여줍니다. 데이터가 어디서 어디로 흐르는가
4 데이터 모델과 mart 설계 L0 Paimon, L1 clean/current, L2 aggregate mart의 역할과 grain을 설명합니다. count를 읽을 기준은 grain이다
5 BI 결과와 해석 대시보드 캡처와 함께 total events, sessions, orders, revenue, review sentiment 등 핵심 지표를 해석합니다. 숫자가 어떤 질문에 답하는가
6 계층별 검증 증거 Kafka message/offset, Flink RUNNING, Paimon row count, Iceberg mart count, Airflow task log, BI query 결과를 연결합니다. 어디까지 정상이라고 말할 수 있는가
7 발견한 리스크와 개선 로드맵 현재 구현의 known gap과 다음 개선 우선순위를 정리합니다. 실무형 포트폴리오의 설계 판단

4. 검증 증거 체크리스트

최종 발표에서 가장 중요한 것은 “돌아갔다”가 아니라 “어떤 증거로 정상이라고 판단했는가”입니다. 아래 중 본인이 직접 확인한 것만 발표 자료에 넣으면 됩니다.

Kafkatopic 3개 존재, message count, offset 증가, key/partition/offset 샘플
Flinkstreaming job 3개 RUNNING, checkpoint/records 관찰, task log
Paimonux_events_bronze, review_current, order_current count 확인
IcebergL1/L2 mart 7개 생성, .snapshots 또는 query log 확인
StarRocks/BIexternal catalog 조회, dashboard 지표와 mart count 비교
AirflowDAG task green, build/query/validate task log, 실패 시 어떤 task에서 멈췄는지

5. 실무 보강 로드맵은 감점 포인트가 아니라 설계 포인트

완성된 프로젝트에도 갭은 남습니다. 중요한 것은 갭을 숨기는 것이 아니라, 무엇을 알고 있고, 왜 지금은 이 수준으로 두었고, 다음에 어떤 순서로 보강할지를 설명하는 것입니다.

항목 현재 상태 실무 보강 방향 발표에서 말할 관점
검증 게이트 위치 현재는 build 후 검증이 많습니다. staging → validate → publish 또는 WAP 구조로 전환합니다. 검증은 서빙 전에 와야 합니다.
이벤트 이력 review/order는 current-state 중심으로 접힙니다. append bronze를 추가해 상태 변경 이력을 보존합니다. current만 있으면 “어떻게 변했나”를 답하기 어렵습니다.
검증 방식 고정 count 기반 검증이 포함되어 있습니다. distinct id, freshness, 입력-출력 관계 같은 불변식 검증으로 확장합니다. 샘플 데이터가 바뀌어도 검증이 살아 있어야 합니다.
StarRocks refresh external catalog cache stale 가능성이 있습니다. DAG에 refresh task를 명시적으로 넣습니다. 조회 계층도 운영 대상입니다.
sentiment 시간 축 전 기간 누적 집계 중심입니다. as_of_date 또는 snapshot 기준 집계를 추가합니다. 추세를 보려면 시간 기준이 필요합니다.
발표에서는 “이것도 안 됐습니다”가 아니라 “현재 MVP의 범위는 여기까지이고, 운영형으로 보강하려면 이 순서로 개선합니다”라고 말하면 됩니다.

6. 하면 좋은 것과 피해야 할 것

하면 좋은 것

  • 본인이 선택한 질문을 발표 첫머리에 명확히 말하기
  • BI 캡처만 보여주지 말고, 어떤 mart에서 나온 지표인지 설명하기
  • 계층별 검증 증거를 한 장으로 모으기
  • count 불일치를 유실로 단정하지 않고 append/upsert 관점으로 해석하기
  • known gap을 우선순위가 있는 개선 로드맵으로 표현하기

피해야 할 것

  • 명령어 실행 캡처만 나열하고 해석하지 않기
  • 공통 파이프라인 전체를 외운 순서대로만 설명하기
  • 모든 지표를 다 보여주려다가 핵심 질문이 흐려지기
  • 남은 갭을 실패처럼 숨기기
  • 새로운 데이터셋으로 급하게 별도 프로젝트를 시작하기
문서 목적: 최종 발표 방향을 학생들에게 설명하기 위한 안내 자료입니다. 프로젝트의 기술 기준은 수업 자료와 GitHub 실습 레포를 기준으로 하며, 이 문서는 발표 구성과 포트폴리오 메시지 정리에 집중합니다.
