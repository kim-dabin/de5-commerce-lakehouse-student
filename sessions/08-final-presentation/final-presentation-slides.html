공통 프로젝트를 개인 포트폴리오로
"이 파이프라인을 실행했습니다"가 아니라
"어떤 질문에 답하고, 어디까지 정상인지 증거로 확인했고,
남은 갭을 어떻게 개선할지 설명할 수 있습니다."
새 데이터를 찾는 게 아니라, 함께 만든 Olist 파이프라인을 내 관점으로 해석·검증해 포트폴리오로 완성한다.
1공통 기반 · 개인화 지점
기반은 같지만 메시지는 달라야 한다. 같은 대시보드를 봐도 질문·해석·발견이 다르면 발표는 충분히 달라진다.

공통 기반 (모두 같음)

  • 데이터: Olist 커머스 + 사용자 행동 이벤트
  • 파이프라인: Kafka → Flink → Paimon → Spark/Iceberg → StarRocks/BI → Airflow

개인화 지점 (각자 다름)

  • 질문 · 지표 해석 · 검증 증거
  • 운영 리스크 · 개선 로드맵 · 나의 관점
2발표 추천 구성 (7장)질문 → 구조 → 증거 → 해석 → 개선
#슬라이드핵심 메시지
1프로젝트 한 줄 소개무엇을 만든 프로젝트인가
2내가 선택한 질문왜 이 프로젝트가 필요한가
3전체 아키텍처데이터가 어디서 어디로 흐르는가
4데이터 모델과 mart 설계count를 읽는 기준은 grain이다
5BI 결과와 해석숫자가 어떤 질문에 답하는가
6계층별 검증 증거어디까지 정상이라 말할 수 있는가
7리스크와 개선 로드맵실무형 포트폴리오의 설계 판단

시간이 짧으면 장수를 줄이되 위 흐름은 유지. 템플릿: PRESENTATION_TEMPLATE.md

3검증 증거 체크리스트직접 확인한 것만
"돌아갔다"가 아니라 "어떤 증거로 정상이라고 판단했는가". 본인이 직접 본 계층만 넣는다. (수집: EVIDENCE_CARD.md)

계층별 증거

  • Kafka topic 3개 · offset · message count
  • Flink 3 job RUNNING · checkpoint
  • Paimon 16,693 / 1,971 / 2,000
  • Iceberg mart 7개 · category_daily 759

품질 · serving · 운영

  • StarRocks/BI external catalog 조회 = mart 일치
  • Airflow DAG success · 어느 task까지
4모두 같으면 안 되는 것

질문 / 지표 / 증거

  • 질문 리뷰를 본 사용자의 구매 전환? 부정 리뷰와 이탈?
  • 지표 어느 테이블·grain에서 나온 숫자인지 설명
  • 증거 어느 계층에서 무엇을 봤는지

리스크 / 개선 / 관점

  • 리스크 count 불일치 · append/upsert · cache stale
  • 개선 검증 게이트 · event bronze · refresh task
  • 관점 내가 가장 중요하게 본 문제 한 문장
5남은 갭은 감점이 아니라 설계 포인트
완성된 프로젝트에도 갭은 남는다. 숨기지 말고 "왜 지금은 이 수준이고, 다음에 어떤 순서로 보강할지"를 말한다.
항목현재보강 방향
검증 게이트 위치build 후 검증 위주staging → validate → publish (WAP)
이벤트 이력review/order는 current-stateappend bronze로 상태 변경 이력 보존
검증 방식고정 count 기반distinct id · freshness · 불변식 검증
StarRocks refreshcache stale 가능DAG에 refresh task 명시

"이것도 안 됐습니다"가 아니라 "MVP 범위는 여기까지, 운영형으로는 이 순서로 보강합니다."

6하면 좋은 것 · 피해야 할 것

하면 좋은 것

  • 첫머리에 내 질문을 명확히
  • BI 캡처에 "어느 mart에서 나온 지표"인지 설명
  • 계층별 증거를 한 장으로 모으기
  • count 불일치를 append/upsert 관점으로 해석
  • known gap을 우선순위 있는 로드맵으로

피해야 할 것

  • 명령 캡처만 나열하고 해석 없음
  • 파이프라인 전체를 암송 순서대로만
  • 지표 다 보여주다 핵심 질문이 흐려짐
  • 남은 갭을 실패처럼 숨기기
  • 새 데이터셋으로 급하게 별도 프로젝트
