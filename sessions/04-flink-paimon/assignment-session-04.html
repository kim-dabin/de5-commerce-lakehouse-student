오늘 만든 흐름

Olist 기반 이벤트를 Kafka topic에서 읽고, Flink SQL streaming job으로 Paimon Bronze/current-state 테이블에 적재합니다.

1. Kafkaux/review/order 이벤트를 topic에 보관
2. Flink SQLKafka source를 계속 구독하는 RUNNING job
3. Paimonappend table과 primary key table에 저장
4. MinIOPaimon warehouse 파일을 object storage에 보관
5. Spark/Jupytersnapshot, files, Iceberg-compatible metadata 관찰
핵심 구분: ux_events_bronze는 행동 발생 사실을 계속 쌓는 append table이고, review_current와 order_current는 같은 key의 최신 상태를 유지하는 current-state table입니다.

먼저 실행할 명령어

아래 명령은 4차시 폴더에서 실행합니다.

cd de5-commerce-lakehouse-student/sessions/04-flink-paimon
git pull
cp .env.example .env  # 이미 있으면 생략 가능
./scripts/check-env.sh

기준 실행 흐름

docker compose -f docker-compose.lite.yml up -d --build

./scripts/reset-olist-kafka-topics.sh
./scripts/reset-olist-paimon.sh
./scripts/run-flink-olist-paimon-streaming.sh

./scripts/produce-olist-ux-events.sh
./scripts/produce-olist-review-events.sh
./scripts/produce-olist-order-events.sh

./scripts/query-olist-paimon.sh
./scripts/query-olist-paimon-ops.sh
주의: reset-olist-paimon.sh 직후에 Jupyter나 query를 실행하면 table not found가 날 수 있습니다. reset은 테이블을 지우는 단계이고, run-flink-olist-paimon-streaming.sh가 테이블을 다시 만들고 streaming job을 올리는 단계입니다.

심화 선택

심화는 선택입니다. 한 가지 이상 해보면 5차시와 6차시에서 운영 관점으로 이어가기 좋습니다.

A. Snapshot / time travel 관찰

./scripts/query-olist-paimon-ops.sh
  • review_current$snapshots 결과에서 APPEND와 COMPACT 흐름을 확인
  • 특정 snapshot id 기준 time travel 결과 확인
  • "문제 직전 정상 기준점"으로 snapshot을 어떻게 쓸 수 있는지 3줄 정리

B. JupyterLab으로 Spark 관찰

docker compose -f docker-compose.lite.yml --profile query up -d --build jupyter
  • http://localhost:8888/lab 접속
  • notebooks/de5-spark-starter.ipynb 실행
  • Paimon $snapshots, VERSION AS OF, Iceberg snapshots 중 하나 캡처
Jupyter가 table을 못 찾으면 먼저 streaming ingest가 실행됐는지 확인하고, 필요하면 커널을 재시작해주세요.

C. MinIO에서 Paimon 파일 보기

  • http://localhost:9001 접속
  • 계정: minioadmin / 비밀번호: minioadmin
  • paimon bucket 아래 warehouse/bronze.db/ 경로 확인
  • snapshot -> manifest -> data file 구조를 본인 말로 설명

D. 같은 데이터 replay 실험

produce 스크립트를 한 번 더 실행한 뒤 count 변화를 관찰합니다.

./scripts/produce-olist-ux-events.sh
./scripts/produce-olist-review-events.sh
./scripts/produce-olist-order-events.sh
./scripts/query-olist-paimon.sh
  • ux_events_bronze는 append라 증가할 수 있습니다.
  • review_current, order_current는 같은 key 기준으로 접혀 row count가 크게 변하지 않을 수 있습니다.
  • 이 차이를 Kafka 중복 방지와 downstream upsert 관점에서 설명해보세요.

자주 만나는 오류

1. Could not find a file io implementation for scheme 's3'

Flink 이미지 안에 paimon-s3 jar가 없는 상태입니다. 최신 코드 pull 후 Flink 이미지를 다시 빌드하세요.

docker compose -f docker-compose.lite.yml exec -T flink-jobmanager \
  bash -lc "ls -1 /opt/flink/lib | grep paimon-s3"

docker compose -f docker-compose.lite.yml stop flink-jobmanager flink-taskmanager
docker compose -f docker-compose.lite.yml build --no-cache flink-jobmanager
docker compose -f docker-compose.lite.yml up -d --force-recreate flink-jobmanager flink-taskmanager

2. Jupyter에서 TABLE_OR_VIEW_NOT_FOUND

reset 후 streaming ingest를 아직 실행하지 않았거나, Spark 커널이 이전 상태를 보고 있을 수 있습니다.

./scripts/run-flink-olist-paimon-streaming.sh
./scripts/query-olist-paimon.sh

# 그래도 안 되면 Jupyter 커널 재시작 후 다시 실행

3. JupyterLab 접속 불가

Jupyter는 기본 profile이 아니라 query profile로 띄웁니다. 8888 포트가 충돌하면 포트를 바꿔 실행하세요.

docker compose -f docker-compose.lite.yml --profile query up -d --build jupyter

# 8888 포트가 이미 사용 중이면
JUPYTER_HOST_PORT=18888 docker compose -f docker-compose.lite.yml --profile query up -d --build jupyter
